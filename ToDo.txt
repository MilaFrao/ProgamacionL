# Etapa 2. Interfaz profesional

Aquí es donde la aplicación empieza a parecer un software comercial.

Por ejemplo:

### Tabla Simplex bonita

En lugar de

```
x1  x2 H1 R1 RHS
```

mostrar

| Base | Cb | x1 | x2 | H1 | R1 | RHS |
| ---- | -- | -- | -- | -- | -- | --- |

como WinQSB.

---

### Resaltar pivote

En cada iteración:

* columna entrante azul
* fila saliente verde
* pivote amarillo

Eso ayuda muchísimo al aprendizaje.

---

### Explicaciones

En lugar de solamente mostrar la tabla:

```
Variable entrante: x2
Variable saliente: H1
```

mostrar algo como:

> x2 entra porque tiene el mayor costo reducido.

> H1 sale porque tiene el menor cociente positivo.

Eso convierte la aplicación en una herramienta educativa.

---

### Navegación

Agregar botones:

```
<<
<
>
>>
```

para recorrer iteraciones.

Como PowerPoint.

---

# Lo que yo haría

Yo me pondría un objetivo muy claro:

> **Construir una aplicación web de Investigación de Operaciones que pueda reemplazar WinQSB para un estudiante universitario.**

Eso le da una dirección al proyecto. En lugar de pensar "¿qué función agrego ahora?", cada mejora se evalúa con una pregunta: **¿esto ayuda a que un estudiante pueda resolver y entender un problema sin salir de la aplicación?**

Con la base que ya tienes (parser, motor Simplex, historial, renderizado de iteraciones y método gráfico), estás más cerca de ese objetivo de lo que parece. A partir de aquí, la mayor parte del trabajo consiste en pulir la experiencia de uso y ampliar el conjunto de algoritmos, no en rehacer la arquitectura.
